> **なぜ合うか:** `ReactionRule` 表はそのまま「特殊ブロックが出会うと何が起きるか」の表です。
> 「水が火を消す」が「ロケット + 爆弾 = 大爆発」に変わるだけで、構造は同じです。

## 部品の対応

| Tempo | マッチ3では |
|---|---|
| `WorldReaction` 格子 | パズル盤面（連鎖 cascade = 格子の広がり） |
| `ReactionRule` | 特殊ブロック組み合わせ表: Trigger=特殊ブロック, Target=周囲ブロック種類, Effect=消去/変換 |
| `DataService<RuleData>` | 表を差し替えるだけで新特殊ブロック追加 = **コード変更なしのライブ運用** |
| `TensionModel` | プレイヤーの圧力（残り手数・目標進捗）で難易度のゴムひも調整 |
| `DirectorBrain` の局面 | 序盤は余裕(BuildUp) → 終盤の山場(Peak) → クリア後の鎮静(Relax) |
| `SeedAction` | `DropGoodPiece`（詰まったら良いブロック） · `AddBlocker`（簡単すぎたら妨害） |

## ルール表の例

要素だけ変えればマッチ3の組み合わせ表になります。

```csharp
// 「ロケットが周囲3種のブロックを列単位で消去」
new ReactionRule {
    Trigger = ElementTag.Fire,             // = Rocket
    TargetMaterials = "Red;Blue;Green",    // 周囲ブロック種類（1欄に ; 区切り）
    Effect = StateEffect.Extinguish,       // = Clear
    Probability = 1f,
};
```

`RuleValidator` がビルド前に「存在しないブロック種類・矛盾するルール」を検出するので、運用中に表を
増やしても安全です。

## ディレクターでゴムひも調整

マッチ3の核心は「ギリギリでクリアさせる」感覚です。ディレクターが緊張度（残り手数の圧力）を見て
条件を撒きます — 詰まったら良いブロックをそっと落とし（`DropGoodPiece`）、簡単すぎたら妨害物を足す。
ただし **前兆**（次に落ちるブロックのプレビュー）を見せ、「運」ではなく「読んで判断」と感じさせます。

## モバイルのチェックポイント

- **表＝ライブコンテンツ。** 新イベントブロックを `RuleData` の1行で追加し、アプリ更新なしでリモートデータ配信。
- **ゴムひもは控えめに。** 露骨すぎると気づかれます。`RiseSpeed`/`FallSpeed` でゆっくり。
- **連鎖は爽快に、上限ははっきり。** 同時連鎖の上限で低スペック端末のカクつきを防ぎます。
